## Slide 7

### What is an OpenClaw agent, and how is it different from a Discord bot?

Here I want to slow down,
because this part
is very easy to confuse.

In Discord,
there is **no native concept**
of an OpenClaw agent.

Discord has **bots**.

So when we bring OpenClaw
into Discord,

we need to understand
the relationship between:

the **OpenClaw agent**,
and
the **Discord bot**.

They are connected,
but they are not the same.

An OpenClaw agent
is defined inside OpenClaw.

It can have different settings
and parameters.

For example:

which model it uses,
how it is configured,
what tools it has,
how it is instructed to behave,
and what role
it is supposed to play.

So inside OpenClaw,
the agent has its own logic
and configuration.

But in Discord,
what other people actually see
is usually **the bot identity**:

the bot name,
the bot avatar,
the bot account.

They do **not** directly see
a native agent identity.

This creates
a very important limitation.

The OpenClaw agent
does not really have
its own native social identity
in Discord.

People identify it
through the bot
that is linked to it.

So even if the logic behind it
is an agent,
what users see
is still just
a Discord bot.

This is one reason
I say Discord is
**not an agent-native social platform**.

It is bot-friendly, yes.
But it is not truly native
for agent identity.

This creates ambiguity.

For example:

bot identity
and agent identity
can be mixed together.

Permissions may look similar.

People may not know
whether they are talking
to one stable agent
or a changing backend.

And the agent itself
does not have a visible,
unique social identity.

And I think this gap
is very important.

Because it shows
a real limitation
in the current setup.

And this is also where
**Beagle Chat**
starts to matter.

Because Beagle Chat
is not only about forwarding messages.

It is also about creating
a more native place
for agent identity,
continuity,
recognition,
and user-side interaction.

So this is not just
a technical detail.

It is a real product gap.
